# Project Plan: Content Performance Dashboard

## Purpose

Content editors currently have engagement metrics (page views, bounce rate, CTR) from Power BI but no signal on how well their articles perform in AI-powered search and chat experiences. This project bridges that gap by adding two AI-performance scores to the existing report, then surfacing everything in a Streamlit dashboard (local or deployed).

The pipeline is designed to be run offline — scores are computed once (or periodically re-run) and stored as JSON files. The dashboard reads from a pre-computed enriched CSV, so there is no live LLM latency during browsing.

## File-by-file reference

### Pipeline scripts (`pipeline/`)

#### `pipeline/fetch_articles.py`
**Purpose:** Scrape and cache article text from learn.microsoft.com.

- Reads a Power BI CSV export and extracts the `Url` column
- For each URL: HTTP GET the page, parse `<main>` / `#main-column` with BeautifulSoup, strip nav/scripts/styles
- Saves clean text to `data/cache/{url_slug}.txt`
- Skips already-cached files unless `--force` is passed
- Adds a 0.5-second delay between requests

**Input:** `--input <csv>` (must have a `Url` column)
**Output:** `data/cache/` directory populated with `.txt` files

---

#### `pipeline/run_ai_readiness.py`
**Purpose:** Grade each cached article across 10 RAG-quality dimensions and assign a High/Medium/Low band.

- Loads the CSV to get the list of URLs
- For each URL, reads the cached text and calls `analyze_dimensions()` from `project-content-improvement/analyze_content.py`
- **Direct mode** (default): runs all 10 dimensions concurrently via `asyncio.gather`
- **Batch mode** (`--batch`): writes one JSONL entry per (article, dimension) pair → submits to Azure OpenAI Batch API → polls until complete
- Counts `len(recommendations)` from each dimension's output; assigns band: High (0–3), Medium (4–8), Low (9+)
- Records `weakest_dimension` (dimension with most recommendations)
- Stores full recommendation text (`evidence`, `action`, `impact`) under `recommendations_by_dimension` for use in HTML reports
- Supports incremental runs: existing scores are loaded and skipped
- `--force`: ignores existing scores and re-grades all articles (triggers LLM calls)

**Input:** `--input <csv>`, `--cache-dir`, `--output`, `--batch`, `--force`
**Output:** `data/scores/ai_readiness_scores.json`

```json
{
  "https://learn.microsoft.com/...": {
    "band": "Medium",
    "total_recommendations": 6,
    "weakest_dimension": "chunk_autonomy",
    "by_dimension": { "heading_hierarchy": 1, "chunk_autonomy": 3, ... },
    "recommendations_by_dimension": {
      "chunk_autonomy": [
        { "evidence": "...", "action": "...", "impact": "..." }
      ]
    }
  }
}
```

Band thresholds are configurable constants at the top of the file (`BAND_HIGH_MAX`, `BAND_MEDIUM_MAX`).

---

#### `pipeline/run_retrievability.py`
**Purpose:** Measure whether the Knowledge Service retrieves each article when asked relevant questions.

Three sub-steps:

**Step 2a — Question generation**
- Calls `send_response_request()` with the `question_generator_article` prompt
- Generates 5 natural-language questions + 5 BM25 keyword queries per article
- Saves checkpoint to `data/scores/questions.json` (re-run with `--skip-qgen` to reuse)

**Step 2b — Knowledge Service queries**
- For each (article_url, question): calls `process_single_question(question, top_k=5)` from `common/knowledge_service.py`
- Uses `ThreadPoolExecutor` (max_workers=10) in batches, with delays between batches
- Checks whether the article's URL path segment appears in any of the 5 returned chunk URLs

**Step 2c — Score calculation**
- `score = (retrieved_count / total_questions) × 100` → integer 0–100
- Supports incremental runs: already-scored URLs are skipped

**Input:** `--input <csv>`, `--cache-dir`, `--questions-file`, `--output`, `--skip-qgen`
**Output:** `data/scores/questions.json` (checkpoint), `data/scores/retrievability_scores.json`

---

#### `pipeline/merge_scores.py`
**Purpose:** Join AI Readiness and Retrievability scores onto the original Power BI CSV.

- Loads the original CSV as a pandas DataFrame
- Left-joins `ai_readiness_scores.json` on the `Url` column, adding:
  - `AIReadiness` — band string (High/Medium/Low)
  - `AIReadiness_WeakestDim` — dimension with most recommendations
  - `AIReadiness_TotalRecs` — total recommendation count
  - `AIReadiness_ByDimension` — JSON string of per-dimension recommendation counts (hidden in dashboard; used internally)
- Left-joins `retrievability_scores.json` on the `Url` column, adding:
  - `Retrievability` — 0–100 score
  - `Retrievability_Retrieved` — number of questions retrieved
  - `Retrievability_Total` — total questions tested
- Articles not yet scored get null values (partial pipeline runs are safe)
- Normalises trailing slashes on URLs before joining to avoid mismatches

**Input:** `--input <csv>`, `--ai-readiness`, `--retrievability`, `--output`
**Output:** `data/enriched_report.csv`

---

#### `pipeline/generate_reports.py`
**Purpose:** Generate self-contained per-article HTML reports from the AI Readiness scores.

- Reads `data/scores/ai_readiness_scores.json` and `config/repo_url_map.json`
- Outputs one HTML file per article to `data/reports/{slug}.html`
- Each report contains:
  - Band chip (High/Medium/Low) and total recommendation count
  - Sign-in notice + "Open in VS Code Web" button (links to private `microsoft-365-docs-pr` repo)
  - "View on Learn" fallback button
  - Dimension breakdown bar chart (weakest dimension highlighted)
  - Full recommendation cards with Evidence / Action / Impact (when `recommendations_by_dimension` is present in the scores JSON; otherwise shows a re-run prompt)
- Report slug is derived from the Learn URL path: `learn.microsoft.com/en-us/a/b/c` → `a_b_c.html`
- Repo map is loaded once per run (not per article) via `load_repo_map()`

**Input:** `--scores`, `--output-dir`, `--url` (optional, for single-article generation)
**Output:** `data/reports/*.html` (one file per scored article)

```bash
python pipeline/generate_reports.py                         # all articles
python pipeline/generate_reports.py --url https://learn.microsoft.com/en-us/...
```

---

#### `pipeline/url_resolver.py`
**Purpose:** Map Learn article URLs to GitHub and VS Code Web URLs.

Two functions:
- `load_repo_map(map_path)` — loads `config/repo_url_map.json` from disk; returns `{}` if absent
- `resolve_github_urls(learn_url, repo_map)` — strips the Learn base prefix, performs longest-prefix matching against the loaded map, returns `github_edit_url`, `vscode_url`, `fallback_url`, and `matched`
- `url_to_slug(url)` — converts a Learn URL to a filesystem-safe slug (shared between `generate_reports.py` and `app.py`)

**Config:** `config/repo_url_map.json` — maps Learn URL prefixes (e.g. `"copilot/microsoft-365"`) to `{ org, repo, branch }`. Unknown prefixes (e.g. `graph/`, `legal/`) fall back to the original Learn URL.

---

#### `pipeline/verify_retrievability.py`
**Purpose:** Diagnostic script for investigating KS retrievability failures on a single article.

Runs the full retrievability check for one URL and prints detailed output — useful for debugging why an article scores low.

### Dashboard (`app.py`)

**Purpose:** Three-tab Streamlit app for browsing and querying the enriched data. Also serves per-article HTML reports inline via query parameters.

**Report routing:** If `?report=<slug>` is present in the URL, the app renders `data/reports/{slug}.html` inline via `st.components.v1.html()` instead of the main dashboard. A "← Back to Dashboard" button clears the query param. Slug validation (allowlist regex + path confinement check) prevents path traversal.

**Sidebar filters** (applied across all tabs):
- Date range (if a date column exists)
- Group / Category
- Topic Type
- AI Readiness band (multi-select)
- Retrievability score range (slider)

**Tab 1 — Data Table**
- Sortable, filterable data table using `st.dataframe` with rich column config:
  - `AIReadiness` — TextColumn with band explanation tooltip
  - `AIReadiness_TotalRecs` — NumberColumn with 10-dimension list tooltip
  - `AIReadiness_WeakestDim` — TextColumn with impact explanation tooltip
  - `AIReadiness_Report` — LinkColumn ("Report ↗") linking to `?report=<slug>` for articles with a generated report
  - `Retrievability` — ProgressColumn (0–100) with methodology tooltip
  - `Retrievability_Retrieved` / `Retrievability_Total` — NumberColumns with tooltips
- Coverage indicator caption: `"Retrievability scored: N/274 articles (X%)"`
- Download button for the filtered view as CSV
- `AIReadiness_ByDimension` column is hidden (internal use only)

**Tab 2 — Portfolio Overview**
- Donut chart: AI Readiness band distribution
- Histogram: Retrievability score distribution
- Scatter plot: Retrievability (x-axis) vs PageViews (y-axis), colored by AI Readiness band
  - Vertical dashed line at score=40 marks the low-retrievability threshold
- Priority table: articles with Low AI Readiness AND Retrievability < 40, sorted by page views

**Tab 3 — Ask the Data**
- Chat input powered by `common/send_openai_request.py` → Azure OpenAI
- Context passed to the LLM includes: full DataFrame schema, per-column statistics, band counts, and up to 5 rows matching any article title/URL mentioned in the question
- Session state preserves chat history for the duration of the browser session

### Config (`config/`)

#### `config/repo_url_map.json`
Maps Learn URL prefixes to GitHub org/repo/branch using longest-prefix matching. Used by `url_resolver.py` to construct "Open in VS Code Web" links in HTML reports.

```json
{
  "copilot/microsoft-365": {
    "org": "MicrosoftDocs",
    "repo": "microsoft-365-docs-pr",
    "branch": "main"
  },
  ...
}
```

Articles whose URL prefix is not in this map fall back to linking to the live Learn article.

## Credentials

Two distinct authentication systems are used:

| System | Credential | Used by |
|--------|-----------|---------|
| Azure OpenAI | `AZURE_OPENAI_ENDPOINT` + `AZURE_OPENAI_KEY` (or `DefaultAzureCredential`) | `run_ai_readiness.py`, `run_retrievability.py` (question gen), `app.py` (chat tab) |
| Microsoft Learn Knowledge Service | `DefaultAzureCredential` scoped to `https://learn.microsoft.com/.default` | `common/knowledge_service.py` — requires `az login` with a Microsoft tenant account |

For Streamlit Cloud deployment, credentials are stored as Streamlit Secrets (TOML format) instead of a `.env` file.

## Design decisions

| Decision | Rationale |
|----------|-----------|
| Band (High/Medium/Low) not numeric score | LLM only flags high-confidence issues; a raw count is more reliable than a hallucinated rating |
| Incremental pipeline runs | Large batches can fail midway; every script resumes from where it left off |
| `--force` flag for re-scoring | Allows full re-grade when prompts or thresholds change, without modifying skip logic |
| Store full recommendation text in JSON | Enables self-contained HTML reports with actionable details without re-running the LLM |
| Static HTML reports, not live rendering | Reports are regenerated by re-running `generate_reports.py`; no LLM latency in the dashboard |
| `?report=<slug>` query-param routing | Allows reports to be linked from the data table without serving a separate web server |
| `load_repo_map()` called once per run | Avoids 273× repeated file reads when generating all reports in a single run |
| 10 questions per article | 5 concise questions test semantic retrieval; 5 BM25 variants test keyword matching |
| Pre-computed CSV, not live scoring | Avoids LLM latency in the dashboard UI; pipeline re-runs can be scheduled overnight |
| Left join on merge | All original CSV rows are preserved even if some articles have not been scored yet |
| `ThreadPoolExecutor` for KS queries | The Knowledge Service handles concurrent requests well; parallelism cuts run time significantly |
| VS Code Web links to private `-pr` repo | Contributors edit the private repo, not the public mirror; sign-in reminder in each report guides authentication |
